# Learning

When creating or updating project-specific Otto rules and commands:

- Be clear: use direct, actionable guidance
- Be concise: avoid repeating platform rules already provided by `ascend.*`
- Be non-duplicative: capture only project-specific practices and validated learnings
- Be non-conflicting: align with existing project rules, validated implementation patterns, and user-approved constraints

Prefer documenting:

- validated architectural patterns
- approved thresholds and guardrails
- project-specific operational constraints
- validated data-quality anomalies with reproducible diagnostics
- platform-safe investigative patterns that ruled out suspected defects

Avoid documenting:

- temporary experiments that were abandoned
- generic platform behavior already covered by Ascend core rules
